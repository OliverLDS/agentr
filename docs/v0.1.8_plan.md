# agentr 0.1.8 Plan

`0.1.8` should focus on lifecycle documentation and proposal persistence polish on top of the `0.1.7` proposal-state refactor.

## Release Goal

Make the workflow lifecycle easier to understand and make proposal objects easier to persist and reuse outside a live `Scaffolder` session.

This release should:

- document the lifecycle in three explicit stages
- expose proposal persistence helpers as supported functions
- document proposal objects as first-class persisted artifacts
- keep the public surface small and avoid introducing public proposal classes yet

This release should not:

- redesign `Scaffolder`
- add new public state classes
- expand execution/runtime scope beyond scaffolding and handoff

## Scope

### 1. Lifecycle documentation

Add or update docs so they clearly separate:

- workflow elicitation
- workflow proposal review and approval
- implementation and extraction handoff

Concretely:

- update `README.md` to use those stage boundaries
- add a focused lifecycle doc for proposal review and persistence
- update `docs/agent_architecture.md` so persisted proposal artifacts are part of the documented design
- update `docs/function_index.md` so proposal persistence helpers are visible

### 2. Proposal persistence surface

Promote these helpers from internal-only behavior to documented package functions:

- `validate_workflow_proposal()`
- `save_workflow_proposal()`
- `load_workflow_proposal()`
- `workflow_proposal_graph_data()`

Also add:

- `print.agentr_workflow_proposal()`

The public goal is not more lifecycle mutation APIs. The goal is to let users:

- persist a proposal after preview
- reload it later for review
- visualize it directly
- validate it before reuse

### 3. Tests

Add or retain coverage for:

- proposal save/load round-trip
- proposal graph export from both proposal object and `Scaffolder + proposal_id`
- proposal print/validation behavior
- README-aligned lifecycle expectations, especially approved workflow versus proposal persistence boundaries

## Definition of Done

`0.1.8` is done when:

- proposal persistence helpers are exported and documented
- lifecycle docs explicitly separate elicitation, review/approval, and handoff
- proposal persistence examples appear in README or lifecycle docs
- the isolated test run passes
- package version/docs align on `0.1.8`
