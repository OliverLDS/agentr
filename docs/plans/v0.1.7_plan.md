# agentr 0.1.7 Plan

`0.1.7` should focus on internal refactoring plus proposal-state formalization without expanding the public surface more than necessary.

## Release Goal

Keep `Scaffolder` as the top-level facade while splitting its internals into clearer modules and making workflow proposals a first-class internal object with explicit lifecycle rules.

This release should:

- preserve the current top-level `Scaffolder` UX
- reduce responsibility concentration in `R/scaffolder.R`
- formalize proposal status values and transitions
- make prompt builders follow one internal convention
- add lifecycle-oriented tests for proposal and handoff flows

This release should not:

- introduce a new public proposal R6 class yet
- redesign the full public API
- bundle long-term persistence/docs work that fits better in `0.1.8`

## Current Repo Reality

The existing code already has the right user-facing pieces for a proposal loop:

- `Scaffolder$propose_workflow()`
- `Scaffolder$list_workflow_proposals()`
- `Scaffolder$get_workflow_proposal()`
- `Scaffolder$approve_workflow_proposal()`
- `Scaffolder$discuss_workflow_proposal()`
- `preview_scaffolder_message()`

But the current implementation still has these weaknesses:

- proposal records are loose lists created inline in `R/scaffolder.R`
- proposal status is effectively ad hoc and currently only uses `pending` and `approved`
- discussion on an approved proposal silently pushes it back to `pending`
- there is no transition helper enforcing valid lifecycle moves
- `Scaffolder` still owns workflow mutation, proposal lifecycle, and other internal helpers in one file
- prompt builders are split across files, but not yet organized around a shared prompt-schema discipline

## Scope for 0.1.7

### 1. Split scaffolder internals into focused files

Create these files and move internal helpers into them:

- `R/scaffolder_workflow.R`
  - decomposition helpers
  - node/edge coercion helpers
  - workflow editing helpers
  - workflow state mutation helpers used by `Scaffolder`
- `R/scaffolder_proposals.R`
  - proposal constructor
  - proposal validation
  - status normalization
  - transition rules
  - proposal save/load helpers
  - proposal graph-data helper
- `R/scaffolder_dispatch.R`
  - action arg normalization
  - action arg validation
  - action reference validation
  - dispatch execution helpers now living in `R/scaffolder_message.R`
- `R/scaffolder_prompts.R`
  - shared prompt-schema helper
  - prompt metadata helpers
  - prompt-builder overview comments and common conventions

Keep exported entry points where users already expect them:

- keep `Scaffolder` defined in `R/scaffolder.R`
- keep exported prompt-builder functions available under their current names
- keep message parse/validate/apply/preview exports unchanged for `0.1.7`

### 2. Formalize proposal objects internally

Add an internal constructor:

- `new_workflow_proposal()`

Return proposal objects with class:

- `agentr_workflow_proposal`

Minimum fields:

- `id`
- `status`
- `source`
- `notes`
- `workflow`
- `discussion_rounds`
- `created_at`
- `updated_at`
- `approved_at`
- `superseded_by`
- `supersedes`
- `rejected_at`

Also add:

- `validate_workflow_proposal()`
- `as_workflow_proposal_summary()`

For `0.1.7`, this can remain an S3-style list object instead of a public R6 class.

### 3. Define fixed proposal statuses and transition rules

Use these status values:

- `pending`
- `under_discussion`
- `approved`
- `superseded`
- `rejected`

Add internal helpers:

- `.normalize_workflow_proposal_status()`
- `workflow_proposal_can_transition()`
- `transition_workflow_proposal()`

Expected transitions:

- `pending -> under_discussion`
- `pending -> approved`
- `pending -> rejected`
- `under_discussion -> approved`
- `under_discussion -> rejected`
- `under_discussion -> superseded`
- `pending -> superseded`

Disallow silently resetting `approved -> pending`.

Instead:

- if new feedback arrives against an approved proposal, create a new proposal or require explicit supersession behavior
- approving a newer proposal should mark the previously latest active proposal as `superseded` when appropriate

### 4. Separate approved workflow, latest proposal, and proposal history

Clarify state boundaries inside `Scaffolder`:

- `self$workflow` remains the current approved workflow
- `self$proposal_log` remains the proposal history store
- add internal helpers to retrieve:
  - current approved workflow
  - latest active proposal
  - full proposal history

Implement these internal helpers:

- `latest_workflow_proposal()`
- `active_workflow_proposals()`
- `proposal_history()`

Behavioral rule for `0.1.7`:

- `build_implementation_prompt(Scaffolder, ...)` should continue to use the approved workflow only
- previewed or discussed proposals must not affect implementation handoff until approved

### 5. Add proposal persistence and graph helpers

Implement the helpers mentioned in the advice, but keep them internal or lightly exported depending on documentation readiness:

- `workflow_proposal_graph_data()`
- `save_workflow_proposal()`
- `load_workflow_proposal()`

`workflow_proposal_graph_data()` should accept either:

- an `agentr_workflow_proposal`
- a proposal id plus `Scaffolder`

Internally it should delegate to `workflow_graph_data(proposal$workflow)`.

Persistence should validate proposal class and status on load.

### 6. Enforce prompt-builder conventions

Keep existing exported names:

- `build_scaffolder_prompt()`
- `build_implementation_prompt()`
- `build_workflow_extraction_prompt()`

For `0.1.7`, standardize the internals rather than renaming public functions.

Add a shared helper in `R/scaffolder_prompts.R`, for example:

- `new_prompt_contract()`

Each prompt builder should define, in code comments and internal payload shape:

- input object type
- target model role
- expected output shape

Apply one convention:

- `build_*_prompt()` is reserved for prompt-generation functions only

Refactor duplicated prompt payload assembly where practical:

- common response-requirement formatting
- common markdown section rendering
- common schema serialization

### 7. Tighten dispatch boundaries around proposal flow

Keep `preview_scaffolder_message()` behavior, but route proposal creation through the new proposal constructor and transition helpers.

Do not let dispatch logic mutate proposal state ad hoc.

Concretely:

- `preview_scaffolder_message()` stores a `pending` proposal
- `discuss_workflow_proposal()` transitions `pending -> under_discussion` if needed
- `approve_workflow_proposal()` validates the transition before promoting workflow
- approving a newer proposal may supersede older active proposals through one shared helper

## File-Level Implementation Sequence

### Step 1. Extract workflow helpers

Move these helper groups out of `R/scaffolder.R` first:

- decomposition helpers
- node/edge coercion
- edit helpers
- metadata append helpers

Goal:

- shrink `R/scaffolder.R` so it mainly contains `Scaffolder` public methods and thin method bodies

### Step 2. Add proposal object helpers

Implement `R/scaffolder_proposals.R` with:

- status constants
- constructor
- validator
- transition engine
- summary conversion
- persistence helpers

Goal:

- replace inline proposal list construction inside `Scaffolder$propose_workflow()`

### Step 3. Rewire `Scaffolder` proposal methods

Update these methods to use the new helper layer:

- `propose_workflow()`
- `list_workflow_proposals()`
- `get_workflow_proposal()`
- `approve_workflow_proposal()`
- `discuss_workflow_proposal()`
- `implementation_spec()`

Goal:

- preserve current top-level signatures where possible
- enforce approved-workflow versus proposal-history separation

### Step 4. Extract dispatch helpers

Move validation and normalization internals out of `R/scaffolder_message.R` into `R/scaffolder_dispatch.R`.

Goal:

- leave `R/scaffolder_message.R` focused on exported prompt/message entry points

### Step 5. Add shared prompt helper

Create `R/scaffolder_prompts.R` and deduplicate prompt metadata/schema assembly.

Goal:

- make prompt builders visibly consistent without changing their public signatures

### Step 6. Expand tests

Add targeted tests before final cleanup so refactors stay pinned to behavior.

## Required Tests for 0.1.7

Add tests covering these lifecycle paths:

- propose -> discuss -> approve
- discuss moves proposal from `pending` to `under_discussion`
- newer proposal approved after older active proposal causes supersession
- approved workflow remains unchanged until proposal approval
- `build_implementation_prompt(scaffolder, ...)` uses approved workflow only
- `preview_scaffolder_message()` stores a proposal without mutating live workflow
- extraction prompt -> inferred workflow preview -> approval path
- proposal save/load round-trip with class and status preserved
- invalid proposal transitions fail with clear errors

Also keep regression coverage for:

- existing `Scaffolder` methods
- existing prompt JSON/markdown outputs
- existing message parse/validate/apply flows

## Documentation Changes Included in 0.1.7

Keep docs limited and tactical in this release:

- update `docs/agent_architecture.md` to mention proposal lifecycle as a distinct internal layer
- add a short prompt-builder overview doc or section comparing:
  - `build_scaffolder_prompt()`
  - `build_implementation_prompt()`
  - `build_workflow_extraction_prompt()`
- update README examples where proposal status semantics change materially

Do not try to fully reorganize all lifecycle docs yet. That belongs in `0.1.8`.

## Backward Compatibility Rules

For `0.1.7`, preserve these public entry points:

- `Scaffolder$new()`
- current exported `Scaffolder` proposal methods
- `build_scaffolder_prompt()`
- `build_implementation_prompt()`
- `build_workflow_extraction_prompt()`
- `apply_scaffolder_message()`
- `preview_scaffolder_message()`

Allowed behavior changes:

- proposal status values become stricter
- proposal discussion no longer rewinds an approved proposal to `pending`
- approval may supersede older active proposals explicitly

Avoid in `0.1.7`:

- renaming exported functions
- requiring users to instantiate a new public proposal class directly
- changing implementation prompt input semantics away from approved workflow

## Definition of Done

`0.1.7` is done when:

- `R/scaffolder.R` no longer carries most proposal and workflow helper internals
- proposal objects have a stable class and validator
- proposal lifecycle transitions are explicit and tested
- approved workflow and proposal history are clearly separated in behavior
- prompt builders use a shared internal contract/convention
- lifecycle tests cover proposal review and implementation handoff boundaries
- existing top-level scaffolder UX remains intact

## Next Version Boundary

Defer these to later versions:

- `0.1.8`: fuller lifecycle docs and proposal persistence/documentation polish
- `0.1.9`: public dedicated proposal/state classes if the surface still grows
