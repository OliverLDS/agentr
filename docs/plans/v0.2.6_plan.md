# agentr 0.2.6 Plan

`0.2.6` should add an interactive review and visualization layer for agent design artifacts while preserving `agentr` as a scaffolding and specification package rather than an execution engine.

## Release Goal

Add a JS-backed human review surface that can be generated from R and used to inspect:

- workflow structure
- knowledge graph structure
- memory and state schema
- human review gates
- structured feedback items

The output should be a portable review artifact:

```text
AgentSpec / WorkflowSpec / KnowledgeSpec
-> review data bundle
-> standalone HTML/JS review page
-> structured feedback JSON
-> validated R feedback object
-> Scaffolder / proposal-state update
```

The browser layer should help humans review and correct design artifacts. It should not run workflow steps, call external APIs, mutate production state, or become a runtime orchestrator.

## Design Principles

### 1. Keep the browser layer review-only

The JS layer may:

- render design graphs
- show memory and state schema
- collect structured comments
- export feedback JSON
- support local visual inspection

The JS layer should not:

- execute workflow nodes
- call provider-specific LLM APIs
- call trading, communication, or data-collection backends
- modify saved R objects directly
- bypass R-side validation

All feedback must return to R as data and pass through validation before it affects `Scaffolder`, proposal state, `AgentSpec`, or any approved artifact.

### 2. Start with standalone HTML before htmlwidgets

The first implementation should generate a standalone HTML file from R.

This is simpler, easier to test, and works without requiring a running Shiny app, Quarto render, or development server.

Target first:

```r
export_design_review_html(agent_spec, "review.html")
```

Then add optional widget-style integration later:

```r
render_design_review_widget(agent_spec)
```

### 3. Use structured feedback, not free text only

Human feedback should be represented as constrained records such as:

```json
{
  "target": "memory_schema",
  "target_id": "agent.memory.state",
  "field": "agent.memory.state",
  "issue": "state names are unclear",
  "suggestion": "separate lifecycle_state from task_state",
  "severity": "medium",
  "status": "open"
}
```

Free-form notes can be included, but every feedback item should have enough structure to be validated, routed, discussed, and later applied.

### 4. Reuse existing graph surfaces

`0.2.6` should build on:

- `workflow_graph_data()`
- `render_workflow_graphviz()`
- `knowledge_graph_from_spec()`
- `knowledge_graph_data()`
- `render_knowledge_graphviz()`

The review page can embed Graphviz SVG output or graph-ready data. The first version should prefer the most reliable path already available in R.

### 5. Keep review data explicit and serializable

The review surface should be powered by a plain list/data-frame bundle that can be inspected and saved independently.

Suggested object:

```r
DesignReviewSpec
```

or, if avoiding a new R6 class in the first pass:

```r
new_design_review_spec()
```

with class:

```r
"agentr_design_review_spec"
```

## Public API Target

### Review bundle helpers

Add:

```r
build_design_review_data()
new_design_review_spec()
validate_design_review_spec()
save_design_review_spec()
load_design_review_spec()
```

Minimum usage:

```r
review <- build_design_review_data(agent_spec)
validate_design_review_spec(review)

```

Supported inputs:

- `AgentSpec`
- `Scaffolder`
- `agentr_workflow_spec`
- `WorkflowProposal`
- `KnowledgeSpec`

For non-`AgentSpec` inputs, missing sections should be represented explicitly as empty review sections rather than causing unnecessary failures.

### HTML export helpers

Add:

```r
export_design_review_html()
design_review_html()
```

Minimum usage:

```r
export_design_review_html(agent_spec, "review.html")
```

`design_review_html()` should return the HTML string without writing it.

`export_design_review_html()` should write the file and return the path invisibly.

Optional arguments:

```r
include_workflow = TRUE
include_knowledge = TRUE
include_memory_schema = TRUE
include_feedback_panel = TRUE
self_contained = TRUE
title = NULL
```

### Feedback helpers

Add:

```r
design_feedback_item()
parse_design_feedback_json()
validate_design_feedback()
save_design_feedback()
load_design_feedback()
apply_design_feedback()
preview_design_feedback()
```

Minimum usage:

```r
feedback <- parse_design_feedback_json("feedback.json")
validate_design_feedback(feedback, review_spec = review)
preview <- preview_design_feedback(scaffolder, feedback)

```

`preview_design_feedback()` should be non-mutating.

`apply_design_feedback()` should mutate only through existing public scaffolder or proposal-state methods where possible.

## Review Data Model

### Top-level review spec

Suggested structure:

```r
list(
  review_id = "review_...",
  generated_at = Sys.time(),
  source = list(
    source_type = "AgentSpec",
    agent_name = "macro-agent",
    task = "..."
  ),
  workflow = list(
    nodes = data.frame(),
    edges = data.frame(),
    graph = list(dot = "...", svg = NULL)
  ),
  knowledge = list(
    items = list(),
    nodes = data.frame(),
    edges = data.frame(),
    graph = list(dot = "...", svg = NULL)
  ),
  memory_schema = list(
    state_spec = list(),
    state_requirements = list(),
    persistent_files = list(),
    schema_constraints = list()
  ),
  interface_schema = list(
    interface_spec = list(),
    interfaces = list()
  ),
  feedback_schema = list(
    allowed_targets = character(),
    allowed_issue_types = character(),
    allowed_severities = character()
  ),
  metadata = list()
)
```

### Allowed feedback targets

Use constrained values:

```r
c(
  "workflow_node",

  "knowledge_item",
  "knowledge_edge",
  "subsystem_assignment",
  "review_gate",
  "implementation_hint",
  "interface_schema",
  "agent_summary"

```

### Allowed issue types

Use constrained values:

```r
c(
  "missing",
  "unclear",
  "incorrect",
  "inconsistent",
  "unsafe",
  "too_broad",
  "too_narrow",
  "duplicate",
  "conflicting_assumption",
  "needs_human_gate",
  "needs_automation_status",
  "needs_trace",
  "implementation_gap"

```

### Allowed severities

Use:

```r

```

### Feedback item shape

Suggested structure:

```r
list(
  id = "fb_001",
  target = "workflow_node",
  target_id = "node_3",
  field = "implementation_hint",

  issue = "The hint does not say which charting package should be used.",
  suggestion = "Specify ggplot2 and expected output dimensions.",
  severity = "medium",
  status = "open",
  source = "human",
  created_at = Sys.time(),
  metadata = list()
)
```

## HTML Review Page Requirements

The standalone page should have three review zones:

1. Workflow graph
2. Knowledge and memory schema
3. Structured feedback panel

### Workflow graph zone

Show:

- node id and label
- inputs and outputs when present
- dependencies
- triggers when present
- checkpoints and human-required gates
- owner and automation status
- linked `knowledge_refs`
- review status and notes

Clicking a node or edge should populate the feedback panel target fields.

### Knowledge and memory zone

Show:

- knowledge items and graph relations
- concepts, conditions, exceptions, causal relations, and rules
- `state_spec`
- `state_requirements`
- persistent files or storage hints
- schema constraints
- interface surfaces when available

Clicking a knowledge item, memory field, or interface field should populate the feedback panel target fields.

### Structured feedback panel

Collect:

- target
- target id
- field
- issue type
- issue text
- suggestion
- severity
- status

The page should maintain an in-browser feedback list and offer:

- add feedback item
- edit feedback item
- delete feedback item
- copy JSON
- download JSON

No server should be required in the first implementation.

## Implementation Order

### Step 1. Review spec data bundle

Implement:

- `new_design_review_spec()`
- `validate_design_review_spec()`
- `build_design_review_data()`

Support `AgentSpec` first.

Then add adapters for:

- `Scaffolder`
- `agentr_workflow_spec`
- `WorkflowProposal`
- `KnowledgeSpec`

Validation should check:

- top-level sections exist
- workflow nodes and edges are data frames
- knowledge graph nodes and edges are data frames
- feedback schema values are known
- source metadata is present

### Step 2. Feedback data model

Implement:

- `design_feedback_item()`
- `validate_design_feedback()`
- `parse_design_feedback_json()`
- `save_design_feedback()`
- `load_design_feedback()`

Validation should reject:

- unknown target values
- unknown issue types
- unknown severities
- missing issue text
- arbitrary code-like method names
- feedback items pointing to impossible target classes

Validation should warn, not error, when a target id is absent from the supplied review spec. This keeps feedback import useful when the reviewed design has changed since export.

### Step 3. Standalone HTML export

Implement:

- `design_review_html()`
- `export_design_review_html()`

The first version should use embedded JSON and plain browser JavaScript.

It should not require:

- Shiny
- Quarto
- a running local web server
- remote CDN resources

### Step 4. Structured feedback application

Implement:

- `preview_design_feedback()`
- `apply_design_feedback()`

Initial routing:

- `workflow_node`: call `Scaffolder$review_node()` or `Scaffolder$discuss_task()` with structured metadata
- `workflow_edge`: record discussion feedback or create an edit proposal if the suggestion is structural
- `memory_schema`: add structured discussion to the agent-spec proposal path
- `knowledge_item`: route to `KnowledgeProposalState` or knowledge discussion helpers
- `review_gate`: update node review or human-required metadata where possible
- `implementation_hint`: propose node-level implementation hint edits

The first version can record most feedback as review/proposal discussion rather than auto-editing every artifact. Auto-editing should be reserved for low-risk fields with clear target ids.

### Step 5. Optional htmlwidgets bridge

After standalone HTML works, evaluate a thin `htmlwidgets` wrapper.

Potential API:

```r
render_design_review_widget(agent_spec)
```

Add this only if it can be done without introducing a heavy dependency burden or making standalone export harder to maintain.

## Pre-v0.2.6 Stabilization: v0.2.5.2

Before implementing `0.2.6`, cut a small stabilization release if any of these are completed:

- tighten `AgentSpec` documentation so patch-release wording does not overstate the historical agent-spec transition
- add explicit `state_spec` and `interface_spec` examples in README or docs, because the review layer will depend on those fields
- add `agent_spec_review_data()` or internal extraction helpers if they are useful preparation for `build_design_review_data()`
- add tests for `AgentSpec` round trips containing `knowledge_spec`, `state_spec`, `interface_spec`, and `autonomy_spec` together
- add one small fixture-style `AgentSpec` builder in tests to avoid duplicating setup across `0.2.6` review tests
- verify generated manual pages for the knowledge-graph functions have clear examples and no stale workflow-only wording
- keep package-level docs aligned with `0.2.5.2` and avoid adding browser-review API references before the API exists

`0.2.5.2` should be a polish and test-hardening release. It should not introduce the JS review page itself unless the implementation is already complete and tested.

## Acceptance Criteria

`0.2.6` is complete when:

1. A user can generate a standalone HTML review page from an `AgentSpec`.
2. The review page shows workflow graph information.
3. The review page shows knowledge graph or knowledge item information when available.
4. The review page shows memory and state schema information when available.
5. The browser page can generate structured feedback JSON without a server.
6. R can parse and validate the feedback JSON.
7. Unknown targets, issue types, and severities are rejected.
8. Previewing feedback does not mutate a `Scaffolder`.
9. Applying feedback records or routes the feedback through existing scaffolder/proposal mechanisms.
10. Existing tests still pass.
11. `R CMD check --no-manual` passes.
12. Documentation clearly states that the review layer is for human inspection and structured feedback, not execution.

## Non-Goals

Do not implement in `0.2.6`:

- a Shiny app
- a hosted review server
- provider-specific LLM calls from the browser
- arbitrary workflow execution from the browser
- direct mutation of saved RDS objects from JavaScript
- account systems, collaboration backends, or remote storage
- full WYSIWYG workflow editing

These can be considered later, but they are outside the current package boundary.
