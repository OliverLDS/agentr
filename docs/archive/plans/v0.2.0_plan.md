# agentr 0.2.0 Plan

`0.2.0` should be the release where `agentr` becomes explicitly agent-spec-first rather than workflow-first.

## Release Goal

Move the package from:

- scaffolding a workflow for a task

to:

- scaffolding an intelligent-agent design for a task

with workflow kept as one component inside that design.

The core conceptual change is to separate two dimensions cleanly:

- subsystem dimension: `RWM`, `PG`, `AE`, `IAC`, `LA`
- workflow dimension: DAG nodes and edges describing task structure

`RWM` is further refined into:

- cognitive layer
- affective layer

## Design Principles

### 1. Keep subsystem and workflow axes separate

Do not collapse Dao et al.'s subsystem model into the DAG.

Instead:

- workflow nodes remain execution-planning artifacts
- each workflow node may be tagged with one or more subsystem owners
- subsystem selection happens before detailed workflow decomposition

### 2. Default to sparse agents

Not all agents need all subsystems.

Defaults should be:

- absent unless selected
- lightweight unless justified
- persistent only when the task truly requires it

Examples:

- simple rule-based automation:
  - strong `AE`
  - moderate `PG`
  - minimal or absent `RWM`, `IAC`, `LA`
- persistent companion or creator agent:
  - meaningful `RWM`
  - maybe affective layer
  - likely some `LA`
  - maybe `IAC`

### 3. Keep `Scaffolder` as the public entry point

`Scaffolder` should remain the facade, but the output target should evolve.

From:

- workflow proposal and workflow spec

To:

- agent design proposal and agent spec

with workflow spec embedded inside agent spec.

### 4. Keep backward compatibility through adaptation layers

`0.2.0` should not delete current workflow functionality.

Instead:

- current workflow spec remains supported
- current proposal and state classes remain supported
- new agent-spec objects wrap and extend those existing artifacts

## Proposed Public Model

### A. Top-level public classes

Add these public classes:

- `IntelligentAgent`
- `AgentSpec`
- `SubsystemSpec`
- `AgentScaffoldState`

Keep these existing classes:

- `Scaffolder`
- `WorkflowProposal`
- `WorkflowProposalState`

### B. Supporting subsystem config classes

Add targeted config classes rather than one giant list blob:

- `RWMConfig`
- `PGConfig`
- `AEConfig`
- `IACConfig`
- `LAConfig`

Within `RWMConfig`, model the inner split explicitly:

- `CognitiveConfig`
- `AffectiveConfig`

These can begin as thin R6 containers with validation and summary methods.

## Public Object Responsibilities

### `AgentSpec`

This should become the main scaffolding artifact for design.

Minimum fields:

- `task`
- `agent_name`
- `summary`
- `subsystems`
- `workflow`
- `state_requirements`
- `interfaces`
- `implementation_targets`
- `metadata`

Responsibilities:

- validate high-level agent design
- expose selected subsystems
- expose workflow spec
- summarize agent shape
- save and load independently

### `SubsystemSpec`

This is the inventory of chosen subsystems and their configs.

Minimum fields:

- `rwm`
- `pg`
- `ae`
- `iac`
- `la`
- `metadata`

Each subsystem field should be either:

- `NULL`
- or a typed config object

Responsibilities:

- validate which subsystems are present
- keep the agent sparse by design
- provide quick summaries like:
  - selected subsystems
  - persistence requirements
  - communication requirements

### `IntelligentAgent`

This is the generated runtime-oriented R6 abstraction.

Minimum fields:

- `id`
- `name`
- `spec`
- `workflow`
- `subsystems`
- `runtime_state`
- `metadata`

Responsibilities:

- hold an `AgentSpec`
- expose selected subsystem configs
- attach current workflow and runtime state
- provide generation hooks for task-specific subclasses later

For `0.2.0`, this class can still be lightweight. The key is to define the abstraction cleanly.

### `AgentScaffoldState`

This should become the new top-level design-state container used by `Scaffolder`.

Minimum fields:

- `approved_agent_spec`
- `proposal_state`
- `workflow_state`
- `metadata`

Responsibilities:

- manage approved agent design state
- manage proposal lifecycle for agent specs
- keep workflow proposal state available as a nested component
- bridge old workflow-first and new agent-first flows

## Proposed Scaffolding Flow

`Scaffolder` should guide humans and reasoning models in this order:

1. task clarification
2. subsystem selection
3. `RWM` layer selection:
   - none
   - cognitive only
   - cognitive plus affective
4. infrastructure and persistence requirements
5. workflow decomposition
6. workflow node subsystem labeling
7. agent-spec review and approval
8. implementation handoff and code skeleton generation

This ordering matters because subsystem footprint should constrain the later workflow and class design.

## New Scaffolder Responsibilities

### 1. Agent-first elicitation

Add internal helpers for:

- subsystem recommendation
- agent-spec proposal building
- workflow-to-subsystem labeling

### 2. Proposal surface

Current workflow proposal handling should be extended, not replaced.

Add:

- `AgentProposal`
- `AgentProposalState`

Or, if you want to keep the surface smaller in `0.2.0`, reuse:

- `WorkflowProposal`
- `WorkflowProposalState`

for workflow only, and introduce:

- `AgentSpec` plus `AgentScaffoldState`

That smaller approach is preferable for `0.2.0`.

### 3. Implementation target generation

Scaffolding should end with an implementation-facing artifact that can generate:

- an `IntelligentAgent` class skeleton
- subsystem stubs
- workflow-node method placeholders

## Prompt and Dispatch Changes

### Prompt layer

Add a new prompt type:

- `build_agent_design_prompt()`

This prompt should ask the reasoning model to:

- identify which subsystems are needed
- justify optional `RWM` layers
- recommend persistence and interfaces
- propose workflow nodes and subsystem ownership

### Dispatch layer

The current dispatch system is workflow-oriented.

For `0.2.0`, extend it with agent-design-safe actions such as:

- `recommend_subsystems`
- `set_subsystem_config`
- `label_workflow_node_subsystem`
- `propose_agent_spec`

Do not expose arbitrary object mutation.

## Data Model Rules

### Workflow node labeling

Each workflow node should gain subsystem metadata such as:

- `primary_subsystem`
- `secondary_subsystems`
- `rwm_layer`

`rwm_layer` should be one of:

- `none`
- `cognitive`
- `affective`
- `cognitive_affective`

Only meaningful when the node touches `RWM`.

### Agent spec sparsity

Subsystem configs should be optional:

- missing subsystem config means subsystem absent
- empty config object should be avoided unless needed

## Migration Strategy

### Backward compatibility

Preserve:

- `Scaffolder$workflow_spec()`
- current workflow proposal methods
- current workflow persistence helpers

Add:

- `Scaffolder$approved_agent_spec()`

Initially, agent-spec generation can wrap existing workflow output and attach subsystem metadata incrementally.

### Transitional behavior

In `0.2.0`:

- workflow-only users should still work
- agent-first users should gain the new public abstractions
- implementation prompts can continue to accept workflow specs, but should also accept `AgentSpec`

## Documentation Work

Docs should explain clearly:

- subsystem model versus workflow model
- when each subsystem is necessary
- why `RWM` is optional for many agents
- why affective layer is rare and task-dependent
- how `agentr` itself is a special self-hosting intelligent agent

Add or update:

- README overview
- architecture doc
- function index
- a subsystem taxonomy doc
- examples for:
  - rule-based automation agent
  - persistent companion or creator agent

## Tests

Add tests for:

- sparse `AgentSpec` with only `AE` and `PG`
- `RWM` config with cognitive only
- `RWM` config with cognitive plus affective
- node subsystem labeling
- `Scaffolder` subsystem recommendation flow
- `AgentScaffoldState` round-trip
- implementation prompt built from `AgentSpec`
- backward compatibility for workflow-only flows

## Recommended Release Boundary

### Must ship in `0.2.0`

- public `AgentSpec`
- public `SubsystemSpec`
- public `IntelligentAgent`
- public `AgentScaffoldState`
- typed subsystem config objects
- subsystem-first scaffolding path
- docs explaining the new model

### Can wait until `0.2.1+`

- richer learning and adaptation machinery
- inter-agent message protocol details
- code generation templates for multiple agent archetypes
- automatic class synthesis from approved specs

## Definition of Done

`0.2.0` is done when:

- `agentr` is documented as an intelligent-agent scaffolding framework, not just a workflow scaffolder
- subsystem selection is first-class in the scaffolding flow
- workflow spec is embedded inside a higher-level agent design model
- the key public classes for agent design are exported and documented
- workflow-only backward compatibility remains intact
- tests pass under the isolated library workflow
